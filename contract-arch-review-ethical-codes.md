# Contract: Architectural Review — ethical-codes-NLP-Index
## Версия: 1.0

---

## SCOPE GUARD: READ-ONLY + REPORT

**Этот контракт запрещает любые изменения файлов.**
Разрешены только: `read_file`, `list_files`, `search_files`.
Результат — один файл `review-report.md`, созданный в корне репозитория.

---

## Задача

Провести архитектурный обзор репозитория `ethical-codes-NLP-Index` по двум направлениям:
1. **Архитектура** — структура проекта, схема данных, связность компонентов
2. **Промпты** — качество инструкций, по которым AI оценивает кодексы

Выходной документ должен быть понятен человеку без технической подготовки.
Никакого жаргона без объяснения. Каждая проблема — на языке «что это значит на практике».

---

## Шаги выполнения

### Шаг 1. Инвентаризация (READ-ONLY)

Прочитать следующие файлы (в указанном порядке):

```
README.md
registry/codes_index.csv
registry/principle_presence_matrix_wide.csv          ← первые 10 строк достаточно
registry/principle_presence_matrix_long.csv          ← первые 10 строк достаточно
registry/principle_index_matrix.csv                  ← первые 10 строк достаточно
docs/topic_registry.md
docs/principles/                                     ← все файлы в папке
prompts/                                             ← все файлы в папке
analysis/yaml/_template                              ← шаблон карточки
analysis/yaml/                                       ← любые 3 готовых YAML (если есть)
```

### Шаг 2. Архитектурный анализ

Оценить по каждому пункту:

**A. Структура репозитория**
- Все ли компоненты из README реально существуют в репозитории?
- Есть ли «осиротевшие» файлы или папки, которые нигде не упомянуты?
- Понятно ли человеку, с чего начать работу с проектом?

**B. Схема данных**
- Совпадают ли колонки в matrix-файлах с тем, что описано в README?
- Консистентна ли шкала оценок (`principle_status`) между файлами?
  - README: 5 значений (`explicit_section | implicit_scattered | brief_mention | absent | unclear`)
  - Что реально встречается в данных? Нет ли отсутствующих категорий?
- Есть ли `document_id` в `codes_index.csv`, для которых нет YAML-карточки, и наоборот?

**C. Связность компонентов**
- Можно ли воспроизвести полный цикл (добавить документ → получить итоговый индекс) по существующим инструкциям?
- Есть ли шаги в «Минимальном рабочем цикле» (README), для которых нет ни инструкции, ни шаблона?

**D. Воспроизводимость результатов**
- Фиксируется ли в данных, какая версия AI и какой промпт использовались при оценке?
  (Поля `Analysis model`, `Date of analysis` в long-матрице — заполнены ли они реально?)
- Если запустить анализ повторно — можно ли сравнить результаты?

### Шаг 3. Анализ промптов

**E. Качество определений принципов (COMP1–COMP10)**
Для каждого принципа из `docs/principles/`:
- Сформулировано ли чёткое определение, что считается «наличием» принципа?
- Разграничены ли принципы между собой (нет ли двух, которые могут описывать одно и то же)?
- Указаны ли примеры фраз или цитат из кодексов, которые соответствуют каждому уровню?

**F. Инструкция для анализа (`prompts/02_single_code_analysis_instruction.md`)**
- Однозначно ли прописано, как выбирать между `explicit_section` и `brief_mention`?
  (Это самое частое место для ошибок AI-классификатора)
- Есть ли инструкция работать с документом на языке оригинала?
- Задан ли формат вывода достаточно жёстко, чтобы AI не мог «отсебятничать» в структуре?

**G. Промпт для синтеза (`prompts/03_synthesis_instruction.md`)**
- Запрещено ли делать выводы «из головы», без опоры на данные?
- Прописано ли, какие файлы брать как источник (wide/long/index)?

## Формат выходного документа

Создать файл `review-report.md` в корне репозитория.

### Структура документа:

```
# Architectural Review: ethical-codes-NLP-Index
Дата: [дата]
Ревьюер: Claude Code

---

## Итоговая оценка
[2–3 предложения: что работает хорошо, на что обратить внимание в первую очередь]

---

## Findings

### 🔴 Критические (нужно исправить до публикации результатов)
[Проблемы, которые могут привести к неверным выводам или невозможности воспроизвести исследование]

### 🟡 Важные (стоит исправить при следующей итерации)
[Проблемы, которые снижают качество или удобство, но не делают результаты ложными]

### 🟢 Улучшения (хорошо бы, но не срочно)
[Вещи, которые улучшат проект, но без них он работает]

---

## Что проверялось
[Список файлов, которые были прочитаны]
```

### Требования к каждому finding:

Каждый пункт в любой из трёх категорий должен содержать:

```
**[Короткое название проблемы]**
Что это: [1–2 предложения без технического жаргона — что именно не так]
Почему важно: [1 предложение — что произойдёт, если не исправить]
Где смотреть: [конкретный файл или папка]
Как исправить: [конкретный шаг — не «улучшить», а «добавить X в файл Y»]
```

**Запрещено:**
- Писать finding без «Как исправить»
- Использовать слова без объяснения: «валидация», «детерминизм», «онтология», «эмбеддинг», «токен»
- Оценивать содержание кодексов этики — только инструменты анализа

## Acceptance Criteria

Контракт считается выполненным, если:

```
grep -c "🔴" review-report.md         # ≥ 0 (секция присутствует)
grep -c "🟡" review-report.md         # ≥ 0 (секция присутствует)
grep -c "🟢" review-report.md         # ≥ 0 (секция присутствует)
grep -c "Что это:" review-report.md   # число = числу всех findings
grep -c "Как исправить:" review-report.md  # число = числу всех findings
grep -c "Что проверялось" review-report.md # = 1
```

- Оценка содержания самих кодексов этики

## Стоп-условия

Остановиться и сообщить оператору, если:
- Папка `prompts/` не существует или пуста — анализ промптов невозможен, продолжить только архитектурную часть
- Матричные файлы пусты — указать это в Итоговой оценке, не делать выводов о схеме данных
- Файлов принципов в `docs/principles/` нет — зафиксировать как 🔴 Critical finding
